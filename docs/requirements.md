# 要件定義書 — Lumina Market

社内リアルタイムオークション「Lumina Market」の要件定義書です。実装前に必ず一度読み、実装中はテーブル名・エンドポイント・ステータス値・エラーcode・WebSocketイベント名で迷ったときの基準として戻ってきてください。

## 背景とストーリー

Lumina社は社員150人に成長し、社内Slackの「譲ります/欲しいです」投稿がタイムラインに埋もれるようになりました。総務チームの発案で、福利厚生として社内オークション/フリマ「Lumina Market」を作ります。盛り上がるように、入札の攻防はWebSocketで**リアルタイム**に見えるようにします。

利用者は全員が同じ会社の社員です。決済や配送は扱わず、落札後は当事者同士がオフィスで受け渡します(取引フローはステータス管理とコメント欄のみ)。その代わり、**同時入札の整合性**と**締切まわりの正確さ**には一切妥協しません。「リアルタイムに動くが、たまに二重勝者が出る」システムは福利厚生でも許されません。

## ペルソナ(利用者)

| 利用者 | ロール | Marketとの関わり |
|---|---|---|
| 一般社員 | member | 出品し、入札し、落札したら受け渡す。全社員がこのロール |
| 総務担当 | admin | カテゴリを整備し、不適切な出品(規程違反・転売等)を停止する。memberの機能もすべて使える |

## 用語集

| 用語 | 意味 |
|---|---|
| 出品(listing) | 売りたい品物の掲載1件。開始価格・入札単位・終了日時を持ち、公開されると入札を受け付けます |
| 開始価格(start_price) | 最初の入札に必要な最低金額。最初の入札は開始価格以上なら有効です |
| 入札単位(bid_increment) | 2回目以降の入札で現在価格に上乗せしなければならない最低額。有効な入札は「現在価格 + 入札単位」以上です |
| 現在価格(current_price) | その時点の最高入札額。単調非減少です(初回入札は開始価格ちょうどがあり得るため据え置き、2回目以降の成立では必ず増加します)。入札が1件もない間は開始価格と同値です |
| 即決価格(buyout_price) | この金額で入札すると即座に落札となる価格(設定は任意)。即決価格以上の入札は即決として扱います |
| 同額先着 | 同じ金額の入札が同時に来た場合、**先にDBの更新に成功した方**が勝ちというルール。後着の同額入札は無効(BID_TOO_LOW)です |
| スナイプ入札 | 終了直前に入札して他者の対抗手段を奪う行為。自動延長で対策します |
| 自動延長 | 終了3分前以降に入札が成立したら終了時刻を3分延長する仕組み。延長は累計30分(10回)が上限です |
| 流札(りゅうさつ) | 終了時刻までに入札が1件もなく、落札者なしで終了すること(listingsのstatusは `ended`) |
| room | WebSocketの配信グループ。出品ごとに1つ(`listing:<id>`)あり、joinしている接続にだけイベントを配信します |
| 閲覧人数(viewer_count) | 出品詳細のroomにjoinしている接続数。盛り上がりの演出用で、厳密なユニークユーザー数でなくて構いません |

## 機能要件(Must)

MoSCoW分析のMust(必須)です。番号はマイルストーンの実装順序とおおむね対応します。

### 1. 認証(会社ドメイン制限)

- サインアップ(`POST /api/auth/signup`)は、メールアドレスのドメインが **`@lumina.example`** の場合のみ許可します。それ以外は `400`(code: `DOMAIN_NOT_ALLOWED`)です。
- ログインするとCookie(HttpOnly)にJWTが設定されます。SNSカリキュラムと同方式です。以降のAPIとWebSocket接続はこのCookieで認証します。
- メール確認(確認リンクの送信)は行いません。社内ドメイン制限がその代わりです(スコープ外を参照)。
- ロールは `member`(既定)と `admin` の2種類。adminは初期データ(seed)で作成します。

### 2. 出品

- タイトル、説明、カテゴリ、画像(最大4枚)、開始価格、入札単位、即決価格(任意)、終了日時を指定して出品できます。
- 金額はすべて**整数(円)**です。開始価格は100円以上、入札単位は10円以上、即決価格は「開始価格 + 入札単位」以上でなければなりません。
- 終了日時は**公開時点から24時間後以上、7日後以内**です。
- 下書き(`draft`)として保存してから公開(`active`)することも、作成と同時に公開することもできます。
- 画像はJPEG / PNG / WebP、1枚5MBまで。ローカルのディスク(`uploads/`)に保存し、URLで配信します(S3化は発展課題 ADV-02)。

### 3. 出品の編集・取消と管理者停止

- 出品者は、**入札が1件も付いていない間**だけ出品を編集・取消(`cancelled`)できます。入札が付いた後の編集・取消は `409`(code: `CONFLICT`)です。
- adminは不適切な出品を停止(`suspended`)できます。停止中は入札できず、一覧の既定表示から外れます。adminは停止を解除(`active` に戻す)できます。

### 4. 一覧・検索

- 出品一覧はキーワード(タイトル・説明の部分一致)、カテゴリ、ステータスで絞り込めます。
- ソートは3種類: 新着順(`newest`)、終了間近順(`ending_soon`)、人気順(`popular` = 入札数の多い順)。
- ページネーション(既定20件、最大100件)を備えます。

### 5. 入札

入札(`POST /api/listings/:id/bids`)の**有効条件**は次のすべてです。1つでも欠ければ拒否します(エラーcodeは [api.md](api.md) の表が正)。

1. 出品のstatusが `active` であること(`AUCTION_NOT_ACTIVE`)
2. 現在時刻(**DBの `now()`**)が `ends_at` より前であること(`AUCTION_ENDED`)
3. 入札者が出品者本人でないこと(`SELF_BID_FORBIDDEN`)
4. 入札者が現在の最高入札者でないこと(`ALREADY_WINNING`)
5. 金額が有効であること(`BID_TOO_LOW`):
   - 入札が1件もない場合(`bids_count = 0`): 金額 ≥ 開始価格
   - 入札がある場合: 金額 ≥ **現在価格 + 入札単位**

同時入札は**高い方が勝ち、同額は先着**です。有効な入札が成立すると `current_price` がその金額に更新され、それまでの最高入札者の入札は `outbid` になります。この判定と更新は競合下でも正しく動くこと(非機能要件を参照)。

### 6. 即決

- 即決価格が設定された出品に対し、**即決価格以上**の金額で入札すると即座に落札(`sold`)となります。`current_price` は即決価格になり、bidsには金額 = 即決価格で記録します(超過分は受け取りません)。
- 即決の成立時は締切処理と同じ後処理(取引作成、bidsの確定、通知、`auction_ended` 配信)を行います。

### 7. リアルタイム配信(WebSocket)

- 出品詳細ページを開くとWebSocketで出品のroomにjoinし、**現在価格・入札件数・入札履歴・残り時間(ends_at)・閲覧人数**の更新をリアルタイムに受信します。
- 最高入札者が入れ替わったとき、抜かれた本人には**即時に `outbid` イベント**を送ります(roomと無関係に本人の接続へ)。
- WebSocketのhandshake時にCookieのJWTを検証します(**SNSカリキュラムのDMと同方式**)。未認証の接続は拒否します。
- イベント名・方向・ペイロードは [api.md](api.md) の「WebSocketイベント一覧」が正です。1文字も違えないでください。

### 8. 自動延長(スナイプ対策)

- 入札成立時、残り時間が**3分以下**(`ends_at - now() ≤ 3分`)なら、`ends_at` を**3分延長**(`ends_at + 3分`)します。
- 延長は1出品につき**累計10回(30分)が上限**です(`extended_count` で管理)。上限到達後は延長せず、入札自体は締切まで受け付けます。
- 延長が起きたらroomに `auction_extended` イベントを配信します。
- 延長の判定・更新は**入札の条件付きUPDATEと同一文内**で行い、締切処理が割り込む隙間を作らないこと(`.claude/skills/bidding-consistency` 参照)。

### 9. 締切と落札

- スケジューラ(10秒間隔を推奨)が `status = 'active' AND ends_at <= now()` の出品を終了処理します。
  - 入札あり: `sold` にし、最高入札を `won`、それ以外を `lost` に確定。取引(trades)を作成し、落札者と出品者にアプリ内通知を送ります。
  - 入札なし: `ended`(流札)にし、出品者に通知します。
- roomには `auction_ended` イベントを配信します。
- **締切後の入札は必ず拒否**されること。締切処理とラスト入札が同時に走っても「締切後の入札が勝つ」ことがあってはなりません(非機能要件を参照)。

### 10. 取引フロー

- 落札が成立すると取引(trade)が自動作成され、`initiated → handover → completed` と進みます(遷移図は [database.md](database.md))。
- 取引画面は**当事者(落札者と出品者)のみ**閲覧・操作できます。第三者は `403` です。
- 当事者間の連絡用コメント欄(受け渡し場所・日時の相談)を備えます。
- `handover`(受け渡し中)へはどちらの当事者でも進められ、`completed`(完了)へは**落札者のみ**が進められます。

### 11. マイページ

- 出品中 / 出品履歴(終了・売却・取消を含む)の一覧
- 入札中の出品一覧(**自分が現在勝っているか負けているか**を表示)
- 落札した出品と取引の一覧
- 未読件数付きの通知一覧(通知の種類は [database.md](database.md) の notifications を参照)

## 機能要件(Should)

- **ウォッチリスト**: 出品をウォッチに追加し、マイページから一覧できます。ウォッチ中の出品の現在価格が更新されたらアプリ内通知(`watch_price_updated`)を受け取ります(自分の入札で更新された場合は通知しません)。
- **代理入札(proxy bid)**: 上限額を登録しておくと、他者に抜かれるたびにシステムが「現在価格 + 入札単位」(上限額まで)で自動的に入札し返します。通常の入札と同じ有効条件・同じ整合性保証に従います。

## 機能要件(Could / 発展)

advancedラベルのissueとして用意しています。必須ではありません。

- **全文検索**: キーワード検索をILIKEからPostgreSQLの `pg_trgm`(または `tsvector`)に置き換え、インデックスが効くことを実行計画で確認します(ADV-01)。
- **画像基盤のS3化**: 画像の保存先をS3 + presigned URL方式(SNSカリキュラムで学んだ方式)に置き換え、バケットをTerraformで定義します(ADV-02。AWSへのapplyは任意)。
- Slack通知(落札のお知らせ等)は設計課題としてのみ扱います(実装スコープ外)。

## 非機能要件

| 項目 | 要件 | 検証方法 |
|---|---|---|
| 同時入札の整合性 | 同一出品への**並列100入札**でも、現在価格の不整合・二重勝者(`winning` が2件以上)・入札単位未満での更新が発生しないこと | 自動テストで並列100入札を実行し、終了後の不変条件(current_price = 最高入札額、winningは常に1件以下、bids_count = 入札行数)を検証します(M2-01) |
| 締切の正確さ | 締切処理とラスト入札が競合しても、**締切後の入札が受理されない**こと。判定はDBの `now()` に統一 | 締切とほぼ同時の入札を再現するテストで「入札が成立して延長される」か「AUCTION_ENDEDで拒否される」のどちらかしか起きないことを確認します(M2-05) |
| 再接続の復元 | WebSocketが切断→再接続しても、状態復元API(`GET /api/listings/:id/state`)で最新の現在価格・履歴・残り時間に追いつけること。取りこぼしたイベントに依存しない | 切断中に入札を進めてから再接続するテストで、表示が最新化されることを確認します(M3-05) |
| セキュリティ | パスワードはハッシュ保存(bcrypt等)。サインアップはドメイン制限。WebSocketは未認証接続を拒否。取引は当事者のみアクセス可 | 未認証WebSocket接続の拒否テスト、第三者の取引アクセスが403になるテストを含めます |

## ユースケース(主要フロー)

### UC-1: 出品から入札成立まで

1. 社員Aがモニターを出品します(開始価格3,000円、入札単位100円、終了48時間後)。公開すると一覧に表示されます。
2. 社員Bが詳細ページを開きます。WebSocketがroomにjoinし、閲覧人数が増えます。
3. Bが3,000円で入札します。入札は1件目なので「開始価格以上」で有効。current_priceは3,000円のままですが、bids_countは1になり、roomに `price_updated` が配信されます。
4. 社員Cが3,100円(現在価格3,000 + 単位100)で入札します。成立するとcurrent_priceは3,100円になり、roomに `price_updated`、Bの接続に `outbid` が届きます。Bには通知(notifications)も残ります。
5. Bが3,100円で再入札しようとすると、必要額は3,200円(3,100 + 100)のため `BID_TOO_LOW` で拒否されます。

### UC-2: 終了間際の攻防と自動延長

1. 残り2分の出品に社員Dが入札します。残り3分以下なので、成立と同時に `ends_at` が3分延長され、roomに `auction_extended` と `price_updated` が配信されます。
2. 社員Eが対抗して入札し、さらに3分延長されます。延長は累計10回(30分)まで繰り返せます。
3. 延長上限に達すると、以降の入札は受け付けられますが `ends_at` は動きません。
4. `ends_at` を過ぎた瞬間以降の入札は、スケジューラの終了処理より先に届いても `AUCTION_ENDED` で拒否されます(判定はDBの `now()`)。

### UC-3: 落札から受け渡し完了まで

1. スケジューラが終了時刻を過ぎた出品を検出し、`sold` に確定します。最高入札は `won`、他は `lost` になり、落札者Eと出品者Aに通知が届きます。roomには `auction_ended` が配信されます。
2. 取引が `initiated` で自動作成されます。EとAは取引画面のコメント欄で「明日の昼、3Fリフレッシュスペースで」と受け渡しを相談します。
3. Aが品物を持参した時点でどちらかが `handover` に進めます。
4. Eが品物を受け取って確認したら `completed` に進めます(completedにできるのは落札者のみ)。取引は終了です。

## スコープ外

以下は本プロジェクトでは**作りません**。issueにも含まれていません。

- 決済(社内の現金・送金のやり取りはシステム外)と配送
- メール送信全般(サインアップのメール確認、落札メール等。通知はアプリ内通知のみ)
- 社外ユーザーの利用、ゲスト閲覧(全ページ認証必須)
- 評価・レビュー、ブロック・通報機能(不適切出品はadminの停止で対応)
- 逆オークション、複数個出品、オークション以外の定額販売モード

> 実務でも「作らないもの」を先に固定すると、issueの受入条件がぶれません。迷ったらこのセクションとissueの受入条件に戻ってください。
